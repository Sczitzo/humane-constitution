# Plain-Language Standard

This project must be readable by people who never went to college. A reader should not need legal training, policy training, software knowledge, or activist vocabulary to understand what the system does and why it matters.

## Main Rule

Write for a tired adult reading on a phone.

Use short words where they work. Use short sentences. Say who does what. Explain the real-world effect before naming the formal rule.

## What To Prefer

| Instead of | Prefer |
| :--- | :--- |
| protocol | system, rule set, or project |
| operative | in force, active, used |
| jurisdiction | place, area, city, region, or government area |
| adjudication | decision, review, or hearing |
| attestation | signed proof, witness statement, or proof |
| supermajority | two-thirds vote, 4-of-5 vote, or large majority |
| quorum | minimum number needed |
| demurrage | carrying cost on accumulated wealth (net-worth demurrage) |
| non-coercion | no one is trapped or forced |
| subsistence / survival floor | basic needs floor |
| constitutional invariant | rule that cannot be quietly changed |
| subsidiarity | keep decisions local when local people can handle them |
| epistemic / methodological | about how facts are measured |
| remediation | fix |
| redress | remedy |
| instrument | tool |
| implementation | how it is put into practice |

Some formal terms still need to exist because the project has exact rules. When a formal term is necessary, define it the first time in plain English.

Example:

> Voice is a civic tool. It can help set priorities, but it can never buy rights, money, housing, food, or power over another person.

## Sentence Rules

- Lead with the person affected.
- Use active voice.
- Keep most sentences under 20 words.
- Do not use Latin phrases.
- Do not use legal filler such as "pursuant to," "herein," "thereof," or "notwithstanding."
- Do not stack abstract nouns: "procedural integrity determination" becomes "a review of whether the process was fair."
- Do not say "shall." Use "must," "may," "cannot," or "will."

## Structure Rules

Every public-facing document should start with:

1. What this is.
2. Who it affects.
3. What changes in real life.
4. What can go wrong.
5. Who checks it.

Deep technical or constitutional documents may keep formal details, but each major section should include a plain-English first paragraph before the precise rule text.

## Tone

Be direct. Do not sound like a court, a grant application, or a software standards body.

The project is allowed to be serious. It should not be hard to read.

## Review Checklist

Before merging a document, check:

- Could a high-school graduate explain the main point after one read?
- Are formal terms explained before they are used heavily?
- Does each rule say who acts, what they do, and what happens next?
- Are there sentences that only sound important because they are abstract?
- Can a person tell how the rule affects food, housing, money, care, voice, or exit?
