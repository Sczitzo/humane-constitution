## Phase 1 Scope

Phase 1 is a structural inventory of the corpus: what files exist, which are exported to the reader, which are excluded, how annexes are numbered, and where naming or linkage inconsistencies appear. Phase 1 does not evaluate substantive constitutional content, evidence quality, or policy soundness. Those are reserved for later phases.

**Deliverables:**
- `docs/audits/00-phase1-audit-manifest.md` — this file
- `docs/audits/01-corpus-inventory.md` — structural inventory and findings

## Known Limitations

1. **No script execution.** `validate_corpus.py` and `export_corpus.py` were not run. Some findings rely on manual cross-reference rather than automated validation output.
2. **Annex content not fully read.** Individual annex files were not read in full; analysis is based on INDEX.md, AH8 table, and file enumeration.
3. **No diff against last known-good corpus.** Without running generate:corpus, any corpus.json / corpus.ts drift from source is not quantified.
4. **Orphan detection is sampling-based.** A complete inbound-link graph was not constructed. Orphan candidates are identified by grep sampling, not exhaustive traversal.
5. **Bash tool partial restriction.** Several grep-pipeline commands were denied mid-audit; some findings substitute Read-tool inspection for scripted analysis.
6. **No semantic content review.** Phase 1 covers structure, naming, and registration only — not constitutional substance.

## Corpus Sharding Strategy

The corpus is split across these export mechanisms in `scripts/export_corpus.py`:

| Mechanism | What it captures |
|---|---|
| `CORE_DOCS` tuple (explicit list) | 52 manually registered paths — governance docs, constitution, architecture, public, simulations, founding, INDEX.md |
| `docs/annexes/ANNEX_*.md` glob | All annex files auto-discovered (currently 43 files) |
| `founding/order/*.md` glob | All founding order docs auto-discovered (currently 5 files) |
| `docs/content/*.md` glob | Reserved path — no files found there currently |

Files not in any of these groups are not exported to the reader corpus. This includes: `docs/PLAIN_LANGUAGE.md`, `docs/review/`, `docs/design/`, `docs/superpowers/plans/`, `README.md` (root), `CODE_OF_CONDUCT.md`, `CONTRIBUTING.md`, `SECURITY.md`, `AGENTS.md`, `CLAUDE.md`.

**Note:** `README.md` (root) IS in `CORE_DOCS` and therefore is exported.

## How Later Phases Should Use Phase 1 Outputs

- **Phase 2 (Linkage integrity):** Use the AH8 gap analysis in `01-corpus-inventory.md` as the starting list of patches not in AH8. Run `validate_corpus.py` to get broken-link output. Cross-check every Annex column entry in AH8 against the file list confirmed here.
- **Phase 3 (Status vocabulary):** Use the findings in `01-corpus-inventory.md` about `Hardening_Queue.md` non-canonical labels as the starting point. Audit every governance doc's status fields against the canonical vocabulary table.
- **Phase 4 (Evidence and claims):** Use the document type map in `01-corpus-inventory.md` to scope the review — focus on Evidence Test Packages, Claims_Evidence_Register, and External_Evidence_Register.
- **Phase 5 (Reader app alignment):** Verify the `featuredPaths` list in `export_corpus.py` against the actual reader UI. Check `READING_PATHS` in Dashboard.tsx against `PATHS` in LandingPage.tsx for sync.

The file `01-corpus-inventory.md` is the canonical output of Phase 1. Later phase auditors should read it before starting.
